1 # MoDeNa {#mainpage}
2 A software Framework for *MO*delling of morphology *DE*velopment of
3 micro- and *NA*nostructures
4 
5 The [MoDeNa project](http://www.modenaproject.eu/) aims at developing,
6 demonstrating and assessing an easy-to-use multi-scale software
7 framework application under an open-source licensing scheme that
8 delivers models with feasible computational loads for process and
9 product design of complex materials. The concept of MoDeNa is an
10 interconnected multi-scale software framework. Four scales will be
11 linked together by this framework namely the nano-, micro-, meso-, and
12 macroscale. As application cases we consider polyurethane foams (PU),
13 which are excellent examples of a large turnover product produced in a
14 variety of qualities and of which the properties are the result of
15 designing and controlling the material structure on all levels of
16 scale, from the molecule to the final product.
17 
18 Multi-scale coupling requires the exchange of information between
19 software instances developed for specific scales in a consistent
20 way. In order to achieve this, generating consistent representations
21 for models and data is necessary. The information exchange is governed
22 by protocols and may occur in two ways, namely:
23 
24 - "forward mapping" (passing information from the microscopic to the
25  macroscopic scale in upward direction)
26 
27 - "backward mapping" (passing information from the macroscopic to the
28  microscopic scale in downward direction)
29 
30 "Forward mapping" is relatively straightforward, while "backward
31 mapping" inevitably requires iteration since changing the operating
32 conditions at the fine level changes the feedback to the coarse level.
33 "Backward mapping" can be realised by "two-way coupling" or by
34 "fitting surrogate models". The first approach usually requires
35 exchange of large amounts of data during runtime that may be expensive
36 either due to the complexity of the data exchange or the computational
37 cost associated with executing the microscopic-scale simulation. In
38 such cases, replacing the microscopic-scale simulation with a
39 surrogate model presents the only viable alternative. This operation
40 inherently constitutes a transfer of data across scales and MoDeNa is
41 unique in that it focuses on this approach.
42 
43 A typical operation sequence starts a macroscopic-scale simulation which
44 instantiates one or more surrogate models. When the validity of a model
45 is violated, a design of experiment operation is triggered. It creates
46 inputs for a set of microscopic-scale simulations. When all experiments
47 are finished, the parameter estimation component is invoked which
48 updates the model parameters. Next, the macroscopic-scale simulation is
49 restarted. It should be noted, that the MoDeNa software framework
50 supports application and model dependencies across multiple scales.
51 
52 The MoDeNa framework handles the communication across scales through
53 recipes and adapters. Recipes perform simulations by executing
54 applications (in-house codes or external software packages such as
55 FOAM, Materials Studio, PC-Saft) for a given set of inputs. Adapters
56 handle the communication with the MoDeNa software framework. Both,
57 recipes and adapters are application specific. Adapters exist as
58 outgoing and incoming adapters. Outgoing adapters are relatively
59 straight forward in that they perform a mapping operation (such as
60 averaging) and communicate the results. The averaging process may have
61 to be started and performed within the application (e.g. for time
62 averaging). However, the results can usually be submitted in a
63 separate process after the simulation is finished. Incoming adapters
64 are more complicated since they usually require to embed surrogate
65 models within the applications.
66 
67 The software framework consists of an orchestrator, a database and a
68 interface library. The orchestrator is based on
69 [FireWorks](http://pythonhosted.org/FireWorks/) and constitutes the
70 backbone of the software framework in that it schedules simulations as
71 well as design of experiments & parameter estimation operations which
72 make up the work-flow of the overall simulation. It is very much like
73 a dynamic work-flow engine, in which the different applications are
74 "orchestrated" to obtain information, analyse and pass it to the other
75 operations. The NoSQL database [MongoDB](http://www.mongodb.org/) is
76 used to store the state of the work-flow as well as the surrogate
77 models together with associated data such as model parameters, data
78 used for parameter estimation, and meta-data.
79 
80 The interface library consists of two parts: A high-level python
81 module providing access to the database as well as design of
82 experiments and regression analysis capabilities by building on
83 [MongoEngine](http://www.mongoengine.org) and
84 [R](http://www.r-project.org/), respectively. The second part is a
85 low-level library providing unified access to the surrogate
86 models. This component is written in C to ensure interoperability
87 across platforms and target applications while providing the
88 computationally efficient model execution required by the
89 applications. The library is loaded as a shared library by the
90 macroscopic-scale applications or as a native python extension by the
91 high-level python module ensuring that all components instantiate
92 identical model implementations. Complex operations such as database
93 access are referred back to the high-level python module using
94 call-back mechanisms.

109 ## Step-by-Step Installation Instructions for Ubuntu:
110 
111 ### Install packages
112 ```
113 user@machine> deb http://cran.rstudio.com/bin/linux/ubuntu trusty/
114 user@machine> sudo apt-get install automake libltdl-dev libltdl7 mongodb \
115  python-pip python-scipy python-rpy2 python-blessings r-base r-base-dev
116 ```
117 
118 ### Install packages (optional)
119 ```
120 user@machine> sudo apt-get install swig
121 ```
122 
123 ### Install pip packages
124 ```
125 user@machine> sudo pip install FireWorks pymongo mongoengine
126 ```
127 
128 ### Install R-libraries (this is done from within the R environment)
129 ```
130 user@machine> R
131 install.packages('lhs', repos='http://cran.cnr.Berkeley.edu/', lib=.libPaths()[1], dependencies=TRUE)
132 install.packages('nlmrt', repos='http://cran.cnr.Berkeley.edu/', lib=.libPaths()[1], dependencies=TRUE)
133 q()
134 ```
135 
136 Press the "n" key and press <ENTER> to exit R without saving the workspace.
137 
138 We are using the Berkley repository. A list of repositories (by nation) can be found at: http://cran.r-project.org/mirrors
139 
140 ### modena interface library & examples (local compilation for now)
141 ```
142 user@machine> cd src
143 user@machine> cmake -DCMAKE_INSTALL_PREFIX:PATH=$HOME .
144 user@machine> make
145 user@machine> make install
146 ```
